# MVP Roadmap

The plan is sequenced so that each phase is shippable and de-risks the next. The hardest engineering — normalizing two very different databases behind one port — is tackled first, because every later feature depends on it. Earlier phases prove the pipeline end-to-end on MySQL; MongoDB is brought to parity once the shape of the abstraction is proven, rather than building both half-way at once.

## Phase 0 — Foundation (this deliverable)

The scaffold is in place: process model, clean-architecture layers, the `IDatabaseConnector` port, repository and service interfaces, the shared cross-process types, the typed IPC contract, the composition root, and the scoring value object. The app boots to an empty shell. Nothing connects to a real database yet, but every seam the rest of the work hangs on is defined and type-checked.

## Phase 1 — MySQL vertical slice (the MVP)

The goal of the MVP is a developer connecting to a real MySQL 8 database, seeing their queries appear, and getting at least one useful index suggestion. This is the phase that proves the product idea.

It covers connection management end to end — the create/edit form, `test()` reporting server version and monitoring capability, saving to SQLite with the password going to the keychain, and edit/delete. It implements the `MySqlConnector` for real: connect, `EXPLAIN FORMAT=JSON`, `listIndexes`, and `collectSince` against `performance_schema`. It implements the analyzer's SQL normalization and fingerprinting, the SQLite history repository with upsert-and-merge, and the collector poll loop with live push events to the UI. On the front end it delivers the live monitoring table and the query-detail screen (raw query, explain plan tree, index used, rows examined/returned, performance score). The recommendation engine ships its MySQL missing-index and composite-index rules. At the end of Phase 1 the core loop — observe, analyze, score, suggest — works for one engine.

## Phase 2 — MongoDB parity

The `MongoConnector` is implemented against the profiler and `system.profile`, with `explain("executionStats")` parsing and `$indexStats` for index usage. The analyzer gains MongoDB command normalization. Because everything above the connector already speaks the normalized vocabulary, the monitoring table, detail screen, scoring, and dashboard work for MongoDB with little to no change — which is the payoff for the port-first design. The recommendation engine adds compound-index suggestions. Reaching parity here validates that the abstraction actually held.

## Phase 3 — Dashboard and insight

With history accumulating for both engines, this phase builds the analytical surface: the metric cards (total, indexed, non-indexed, slow, average time), the four rankings (slowest, most-executed, full scans, poor selectivity), and the Recharts time series of query latency and volume. Filtering and search over query history land here too, along with the monitoring settings screen (slow threshold, poll interval, retention, auto-explain).

## Phase 4 — Recommendation depth

The remaining recommendation rules are added: MySQL redundant-index detection (prefix and duplicate indexes) and MongoDB unused-index detection over the observation window. Suggestions become dismissible and gain estimated-impact figures. This is where the tool moves from "shows you problems" to "tells you exactly what to change and what it will save."

## Phase 5 — Polish and packaging

Final UI refinement against the TablePlus/Compass/Datadog reference, empty and error states, keyboard navigation, the test suite (the pure analyzer and recommendation rules are the highest-value targets), and `electron-builder` packaging for Windows, macOS, and Linux with auto-update.

## Phase 6 — Workload pattern insights

After the single-query recommendation loop is stable, the app starts looking for performance problems that only appear across a burst or request-shaped workload. The headline feature is deterministic N+1 query detection: many individually-fast, similarly-shaped point lookups that add up to a slow page, API request, or background job even when each query appears healthy in isolation.

The analyzer groups captured queries by fingerprint, connection, collection/table target, and short time windows, then flags repeated lookup bursts where the same normalized query runs many times with different literal values. These insights are scored by total cost rather than per-query latency: execution count, window duration, cumulative time, rows examined, rows returned, and whether the query is indexed. For SQL workloads, the first rules identify patterns such as repeated `WHERE id = ?` or `WHERE foreign_key = ?` lookups that could become `IN (...)`, joins, preloading, or eager loading. For MongoDB, equivalent rules identify repeated document lookups that could become `$in`, `$lookup`, denormalization, or prefetching depending on the shape.

The UI adds a workload-insights surface next to index recommendations: "84 similar queries in 2.3s," "720ms cumulative time," "indexed but repeated," and "batching candidate." These findings are deliberately separated from missing-index suggestions so the app does not recommend another index when the real issue is query orchestration. This phase also lays the groundwork for future request correlation by keeping the first version useful without framework instrumentation, then allowing optional `traceId` or `requestId` metadata later.

## Phase 7 — PostgreSQL parity

PostgreSQL becomes the third supported engine after the app is polished and packaged for the MySQL/MongoDB core. The goal is not just "another SQL connector," but a high-quality PostgreSQL adapter that uses the engine's richer planning and statistics surface to deepen the deterministic product.

The `PostgresConnector` implements the same connector port as the existing engines: connection testing, `EXPLAIN (FORMAT JSON)` parsing, index inventory, and query collection through `pg_stat_statements` when available. The adapter normalizes PostgreSQL plan nodes such as `Seq Scan`, `Index Scan`, `Index Only Scan`, `Bitmap Index Scan`, `Bitmap Heap Scan`, `Nested Loop`, `Hash Join`, `Merge Join`, and `Sort` into the shared execution-plan vocabulary, while preserving PostgreSQL-specific evidence for the detail screen. Optional `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)` support is added for explicit per-query inspection so the app can distinguish cached CPU-heavy work from real I/O pressure without making continuous monitoring invasive.

The analyzer and recommendation engine gain PostgreSQL-specific rules: missing composite indexes for `WHERE` plus `ORDER BY`, unused and duplicate indexes from catalog statistics, expression-index suggestions for predicates like `LOWER(email)`, partial-index suggestions for stable filtered workloads, GIN advice for JSONB containment and array queries, and warnings when a function or cast prevents index usage. The UI gains PostgreSQL capability guidance around `pg_stat_statements`, required permissions, optional `BUFFERS` analysis, and extension-dependent recommendations such as `pg_trgm`.

## Phase 8 — AI query optimization

Everything through Phase 7 is deterministic: rule-based scoring, rule-based index suggestions, workload-pattern insights, no model in the loop. Phase 8 adds a language-model layer on top of that foundation, turning "here is the index you're missing" into "here is the better query, here is what your plan is actually doing, and here is the schema change worth considering." The deterministic engine remains the source of truth; the model only ever consumes the analysis the app already produced and explains or rewrites on top of it, so a wrong or unavailable model degrades the feature to the existing rule-based experience rather than breaking it.

The phase delivers three capabilities, all driven from the query-detail screen and all optional. The first is **query rewriting**: given the normalized query, its execution plan, and the index inventory, the model proposes a more efficient equivalent — dropping `SELECT *`, reshaping a join, removing a function wrapped around an indexed column, replacing an `OR` with a `UNION`, adding a covering projection in MongoDB — and explains in plain language why the rewrite helps and what to verify. Each suggestion is presented as a diff against the original with a one-line rationale, never auto-applied. The second is **plan explanation and richer index advice**: the model narrates the execution plan in natural language and enriches the existing rule-based index suggestions with the *reasoning* behind them, so a developer understands not just that a composite index on `(status, created_at)` is recommended but why that column order matters for their specific predicate and sort. The third is **schema and data-model advice**: working from the query patterns observed over the monitoring window rather than a single query, the model suggests normalization fixes, tighter column types, and partitioning or sharding directions, clearly labeled as higher-effort, lower-confidence guidance to be weighed rather than applied blindly.

The integration is built behind a provider port so the privacy posture is preserved and the user chooses the trust boundary. An `ILlmProvider` interface abstracts the model the same way `IDatabaseConnector` abstracts the database, with two adapters: a **local adapter** (Ollama / an OpenAI-compatible localhost endpoint) that keeps the no-cloud promise intact, and a **cloud BYO-key adapter** (OpenAI / Anthropic) whose API key is stored in the OS keychain through the existing `ISecretStore` port, exactly as database passwords already are — never written to SQLite. AI features ship **disabled by default**; nothing leaves the machine until the user opts in, picks a provider, and, for the cloud path, enters a key. A settings screen exposes provider choice, model name, endpoint, and an explicit notice of what query text and schema metadata would be sent to a third party on the cloud path. Suggestions are cached against the query fingerprint so the same query is not re-sent on every view, and the prompts deliberately send normalized query shapes and plan/index structure rather than literal parameter values, keeping sensitive data out of the request.

## Sequencing rationale

Connection management and the MySQL connector come first because nothing is testable without a live source of queries. MySQL leads because `performance_schema` digests give clean aggregates with the least collection complexity, so the analyzer and storage can be validated before the messier profiler integration. The dashboard waits until real history exists, since building charts against empty tables teaches nothing. Redundant- and unused-index detection come late because they need a meaningful observation window to be trustworthy. Packaging and the bulk of automated tests land before broader workload analysis, once the MySQL/MongoDB interfaces have stopped moving. Workload pattern insights follow because they turn stored history into higher-level diagnosis: the app can catch N+1 and repeated-lookup problems that are invisible when each query is judged alone. PostgreSQL follows because it expands the core deterministic product to the most important missing developer database while exercising the connector abstraction with richer SQL planning features. AI optimization is sequenced last because it is strictly additive — it consumes the normalized analysis, workload insights, plans, and recommendations that the deterministic engine already produces, so it cannot be built (or trusted) until the multi-engine outputs are stable and validated. Placing it after PostgreSQL also keeps the model an optional, opt-in layer rather than a dependency of the core loop: the product is fully useful with the AI features turned off, and Phase 8 only deepens the advice for users who choose to enable it.
